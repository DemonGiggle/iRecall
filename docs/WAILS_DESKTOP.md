# Wails Desktop Mapping

## Purpose

This document defines how the desktop implementation should map to the shared product contract in [UI_DESIGN.md](UI_DESIGN.md) and the richer-client redesign in [DESKTOP_WEB_UX.md](DESKTOP_WEB_UX.md).

`UI_DESIGN.md` remains the shared behavioral contract.

`DESKTOP_WEB_UX.md` is the source of truth for desktop/web interaction hierarchy, page flow, and modal reduction.

This file focuses on:

1. the current Wails application structure
2. boundaries between the Go backend and frontend shell
3. platform-specific behavior such as file dialogs and bootstrap state

## Directory Structure

The desktop implementation lives under `desktop/`.

```text
iRecall/
├── app/
│   ├── app.go
│   └── app_test.go
├── frontend/
│   ├── package.json
│   ├── tsconfig.json
│   ├── vite.config.ts
│   ├── index.html
│   └── src/
└── desktop/
    ├── main_wails.go
    ├── wails.json
    └── README.md
```

Rules:

1. keep desktop-specific code under `desktop/`
2. keep product logic in `core/`
3. never move Windows-specific UI logic into `tui/`
4. never move Wails transport concerns into `core/`

## Architecture

### Shared domain

`core/` remains the shared application layer.

It owns:

1. quotes
2. settings
3. user profile
4. recall orchestration
5. import/export semantics
6. tag extraction

### Desktop application layer

`app/` is the shared desktop/web application service.

It currently:

1. open the engine
2. load bootstrap state
3. expose frontend-friendly methods
4. translate file-based import/export to UI-level commands
5. save and load the local profile
6. run the full recall workflow and return the completed result

It should not:

1. contain Wails widget code
2. contain page layout code
3. duplicate domain rules from `core/`

### Frontend shell

`frontend/` owns:

1. shell layout
2. desktop page composition
3. dialog rendering
4. command invocation against the shared application service

It should follow `UI_DESIGN.md` and `DESKTOP_WEB_UX.md` rather than the terminal layout literally.

## Page Mapping

### Recall

Desktop mapping:

1. top question field
2. keywords summary line
3. main answer region
4. evidence/reference quote region

Recommended desktop adaptation:

1. treat this page as the primary workspace
2. use a splitter for response and evidence panes
3. move quote-management actions out of the main Recall toolbar and into evidence context
4. support both keyboard and pointer interactions

### Quotes

Desktop mapping:

1. main list view
2. quote actions in toolbar, context menu, or shortcut layer
3. import/export via native file dialogs

The desktop UI should improve ergonomics with:

1. search + list-detail structure
2. contextual actions in the detail pane
3. explicit select mode for bulk actions
4. native multi-select interactions only when the user enters that mode

But it must preserve:

1. current-row fallback when nothing is multi-selected
2. source attribution for foreign quotes

### History

Desktop mapping:

1. history summary list
2. modal detail view for one saved recall session
3. save-as-quote and rerun actions from the detail view
4. multi-select delete support in the summary list

Current implementation note:

1. the shared desktop/web frontend still opens history detail in a blocking modal rather than a persistent split pane

### Settings

Desktop mapping:

1. connection section
2. retrieval section
3. personalization section
4. collapsed security and advanced sections
5. explicit test-connection and fetch-models actions

Use native controls where appropriate:

1. text fields
2. toggle switch or checkbox for HTTPS
3. combo box for model selection after fetch

## Modal Mapping

### Set name

Use a blocking dialog on startup if the display name is missing.

Keep:

1. explanatory copy
2. required non-empty validation
3. non-dismissible behavior until valid save

### Add / Edit quote

Use a drawer or large side sheet by default.

Keep:

1. shared add/edit form
2. explicit save
3. refine action
4. refine comparison preview before apply

### Delete confirmation

Use a confirmation dialog.

Keep:

1. explicit destructive confirmation
2. quote summary
3. irreversible copy

### Share / Export

Use a native save-file picker with a compact export summary rather than a raw text path field.

Keep:

1. preview or summary of what is being exported
2. manual file-based transfer model
3. actual export payload generated by the backend

### Import

Use a native open-file picker with a validation/result summary rather than a raw text path field.

Keep:

1. file-based import
2. post-import result summary
3. reload of the quote library after successful import

## Backend API Shape

The current desktop backend exposes task-level methods rather than raw storage APIs.

The implemented surface includes:

1. `BootstrapState()`
2. `ListQuotes()`
3. `AddQuote(content)`
4. `SaveRecallAsQuote(question, response, keywords)`
5. `RefineQuoteDraft(content)`
6. `UpdateQuote(id, content)`
7. `DeleteQuotes(ids)`
8. `PreviewQuoteExport(ids)`
9. `SelectQuoteExportFile()`
10. `ExportQuotesToFile(ids, path)`
11. `SelectQuoteImportFile()`
12. `ImportQuotesFromFile(path)`
13. `GetSettings()`
14. `SaveSettings(settings)`
15. `FetchModels(settings)`
16. `GetUserProfile()`
17. `SaveUserProfile(name)`
18. `RunRecall(question)`
19. `ListRecallHistory()`
20. `GetRecallHistory(id)`
21. `DeleteRecallHistory(ids)`
22. `AuthStatus()`
23. `SetupPassword(password, confirm)`
24. `Login(password)`
25. `ChangePassword(current, next, confirm)`

When extending it, prefer task-level methods over low-level transport methods.

Good examples:

1. `RunRecall(question)`
2. `ExportQuotesToFile(ids, path)`
3. `ImportQuotesFromFile(path)`

Avoid leaking storage implementation details into the frontend.

## Windows-specific guidance

Recommended desktop behavior:

1. use native open/save dialogs for import/export
2. keep JSON payload visibility behind an advanced disclosure
3. allow window resizing freely
4. keep the title bar greeting visible but secondary
5. use persistent split-view layouts where possible

Avoid:

1. adding unrelated dashboard widgets
2. splitting the app into many extra top-level sections
3. changing quote sharing from manual files without a deliberate product decision

## Implementation sequence

Recommended order:

1. bootstrap shell from `app.App`
2. implement user-profile startup gating
3. implement Quotes page CRUD
4. implement import/export with native file pickers
5. implement Recall page
6. implement Settings page
7. add desktop polish

## Maintenance rule

If the desktop client differs from `UI_DESIGN.md` intentionally, document that decision here.

If not, this document should stay thin and act only as a platform mapping layer.
