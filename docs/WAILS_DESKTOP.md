# Wails Desktop Mapping

## Purpose

This document defines how the current desktop implementation maps to the shared product contract in [UI_DESIGN.md](/home/gigo/workspace/iRecall/docs/UI_DESIGN.md).

`UI_DESIGN.md` is the source of truth for product behavior.

This file focuses on:

1. the current Wails application structure
2. boundaries between the Go backend and frontend shell
3. platform-specific behavior such as file dialogs and bootstrap state

## Directory Structure

The desktop implementation lives under `desktop/`.

```text
desktop/
├── backend/
│   ├── app.go
│   └── app_test.go
├── frontend/
│   ├── package.json
│   ├── tsconfig.json
│   ├── vite.config.ts
│   ├── index.html
│   └── src/
├── main_wails.go
├── wails.json
└── README.md
```

Rules:

1. keep desktop-specific code under `desktop/`
2. keep product logic in `core/`
3. never move Windows-specific UI logic into `tui/`
4. never move Wails transport concerns into `core/`

## Architecture

### Shared domain

`core/` remains the shared application layer.

It owns:

1. quotes
2. settings
3. user profile
4. recall orchestration
5. import/export semantics
6. tag extraction

### Desktop application layer

`desktop/backend` is the desktop-facing application service.

It currently:

1. open the engine
2. load bootstrap state
3. expose frontend-friendly methods
4. translate file-based import/export to UI-level commands
5. save and load the local profile
6. run the full recall workflow and return the completed result

It should not:

1. contain Wails widget code
2. contain page layout code
3. duplicate domain rules from `core/`

### Frontend shell

`desktop/frontend` owns:

1. shell layout
2. desktop page composition
3. dialog rendering
4. command invocation against the backend

It should follow `UI_DESIGN.md` rather than the terminal layout literally.

## Page Mapping

### Recall

Desktop mapping:

1. top question field
2. keywords summary line
3. main answer region
4. evidence/reference quote region

Recommended desktop adaptation:

1. use a splitter for response and reference panes
2. keep focus distinction visible
3. support both keyboard and pointer interactions

### Quotes

Desktop mapping:

1. main list view
2. quote actions in toolbar, context menu, or shortcut layer
3. import/export via native file dialogs

The desktop UI may improve ergonomics with:

1. native multi-select list interactions
2. toolbar buttons for add, import, export, edit, delete

But it must preserve:

1. current-row fallback when nothing is multi-selected
2. source attribution for foreign quotes

### Settings

Desktop mapping:

1. provider form section
2. search configuration section
3. explicit save affordance
4. fetch-models action

Use native controls where appropriate:

1. text fields
2. toggle switch or checkbox for HTTPS
3. combo box for model selection after fetch

## Modal Mapping

### Set name

Use a blocking dialog on startup if the display name is missing.

Keep:

1. explanatory copy
2. required non-empty validation
3. non-dismissible behavior until valid save

### Add / Edit quote

Use a dialog or drawer.

Keep:

1. shared add/edit form
2. explicit save
3. refine action
4. refine comparison preview before apply

### Delete confirmation

Use a confirmation dialog.

Keep:

1. explicit destructive confirmation
2. quote summary
3. irreversible copy

### Share / Export

Use a native save-file picker rather than a raw text path field.

Keep:

1. preview or summary of what is being exported
2. manual file-based transfer model
3. actual export payload generated by the backend

### Import

Use a native open-file picker rather than a raw text path field.

Keep:

1. file-based import
2. post-import result summary
3. reload of the quote library after successful import

## Backend API Shape

The current desktop backend exposes task-level methods rather than raw storage APIs.

The implemented surface includes:

1. `BootstrapState()`
2. `ListQuotes()`
3. `AddQuote(content)`
4. `RefineQuoteDraft(content)`
5. `UpdateQuote(id, content)`
6. `DeleteQuotes(ids)`
7. `PreviewQuoteExport(ids)`
8. `ExportQuotesToFile(ids, path)`
9. `ImportQuotesFromFile(path)`
10. `GetSettings()`
11. `SaveSettings(settings)`
12. `FetchModels(settings)`
13. `GetUserProfile()`
14. `SaveUserProfile(name)`
15. `RunRecall(question)`

When extending it, prefer task-level methods over low-level transport methods.

Good examples:

1. `RunRecall(question)`
2. `ExportQuotesToFile(ids, path)`
3. `ImportQuotesFromFile(path)`

Avoid leaking storage implementation details into the frontend.

## Windows-specific guidance

Recommended desktop behavior:

1. use native open/save dialogs for import/export
2. use copy buttons where JSON payload visibility is useful
3. allow window resizing freely
4. keep the title bar greeting visible
5. preserve the three top-level pages

Avoid:

1. adding unrelated dashboard widgets
2. splitting the app into many extra top-level sections
3. changing quote sharing from manual files without a deliberate product decision

## Implementation sequence

Recommended order:

1. bootstrap shell from `desktop/backend.App`
2. implement user-profile startup gating
3. implement Quotes page CRUD
4. implement import/export with native file pickers
5. implement Recall page
6. implement Settings page
7. add desktop polish
